# 技术边界与异常分析 (Technical Understanding & Bug Analysis)

## 1. 前后端 API 数据交互解析
在电商详情页的渲染中，前端依赖后端 API 提供的数据字典。基于底层系统架构的规范，梳理出以下核心业务映射规则：

### 1.1 价格精度处理
* **技术现象**：API 返回的价格字段为 `price: 19900`。
* **业务映射**：底层数据库为规避浮点数计算造成的精度丢失，金额单位统一设定为“分”。前端在渲染视图时，需强制执行 `/100` 的换算，并拼接货币符号，最终向用户展示为 `¥199.00`。

### 1.2 数据模型缺失评估
在评估当前 API 结构时，发现缺失以下商业闭环必要字段，需在后续迭代中补充：
* **富文本详情 (`description`)**：用于承载长图文，辅助消费者决策。
* **运费规则 (`freightRule`)**：决定订单结算时的最终应付金额。
* **限购阈值 (`purchaseLimit`)**：用于防范恶意刷单或执行特殊促销策略。

## 2. 交易阻断逻辑与状态机 (State Machine)

### 2.1 结算拦截守卫逻辑
在提交订单的伪代码级校验中，系统必须构筑多道防线：
* **售卖状态校验**：若 `productStatus !== 'ON_SALE'`，强制阻断。
* **物理库存校验**：若 `skuStock <= 0`，判定为售罄，强制阻断。
* **并发超卖校验**：若购买意向数大于可用库存 (`buyCount > skuStock`)，强制阻断。
* **补充防线（规划中）**：需补充账号黑名单拦截、收货地址合法性校验。

### 2.2 订单全生命周期流转
订单状态机定义了业务实体在不同阶段的法定表现形式，核心流转路径如下：
* **正向交易流**：`PENDING_PAYMENT` (待支付) $\rightarrow$ 支付成功 $\rightarrow$ `PAID` (已支付) $\rightarrow$ 录入物流 $\rightarrow$ `SHIPPED` (已发货) $\rightarrow$ 用户签收 $\rightarrow$ `COMPLETED` (已完成)。
* **正向异常流**：`PENDING_PAYMENT` (待支付) $\rightarrow$ 触发 30 分钟支付超时或主动放弃 $\rightarrow$ `CANCELLED` (已取消) $\rightarrow$ **执行库存回滚**。
* **逆向售后流**：`PAID` / `SHIPPED` $\rightarrow$ 发起退款 $\rightarrow$ `REFUNDING` (退款中) $\rightarrow$ 财务审批 $\rightarrow$ `REFUNDED` (已退款)。

## 3. 典型并发 Bug 深度排查

### 3.1 异常场景重现
* **QA 提报问题**：前端商品详情页显示库存为 1，但用户点击“提交订单”时，系统报错“库存不足”。

### 3.2 根因排查剖析
这种情况通常并非代码运算错误，而是高并发架构下的正常现象，排查重点在于数据一致性：
* **前端缓存延迟 (Cache Delay)**：为承载瞬时流量，详情页的库存数据常读取自 Redis 等缓存组件。用户看到的“1”可能是未刷新的旧切片数据，底层 MySQL 数据库中的实际可用库存已归零。
* **并发抢单时间差 (Race Condition)**：用户 A 停留在详情页期间，用户 B 已率先触发了结算接口并成功锁定了最后 1 件库存。
* **后端强校验机制**：订单提交 API 执行了正确的二次校验。这证明了**“后端校验才是业务最终事实来源 (Single Source of Truth)”**这一核心架构原则。

### 3.3 产品侧优化方案
在此类边界场景下，应重点优化前端交互体验：当系统驳回结算请求抛出“库存不足”提示的同时，前端需静默刷新接口，强制拉取并展示最新的“已售罄”状态，消除用户的困惑与不信任感。
