# MVP 功能列表与优先级划分

## 1. 核心定义与定级标准

* **MVP (Minimum Viable Product)**: 最小可行性产品，旨在用最低的成本、最少的核心功能跑通完整的商业交易闭环。
* **优先级模型**:
  * **P0 (Must-have)**: 核心必做。缺少该功能则基础交易链路（浏览-下单-支付-发货）将彻底瘫痪。
  * **P1 (Should-have)**: 重点提升。不阻断交易主干流程，但能显著提升前端用户体验或后台运营效率。
  * **P2 (Nice-to-have)**: 锦上添花。非紧急需求，可在后续迭代中逐步完善。

## 2. 消费者端 (Customer Side) 功能矩阵

本模块旨在保障 C 端用户能够顺利完成基础购买全流程。

### 2.1 账户体系
* **核心功能**: 登录 / 注册
* **优先级**: P0 (核心必做)
* **业务原因阐述**: 必须具备唯一的用户身份标识，系统才能生成并绑定关联的订单记录与支付状态。

### 2.2 商品导购
* **核心功能**: 商品列表
* **优先级**: P0 (核心必做)
* **业务原因阐述**: 提供基础的商品信息展示入口，是触发交易链路的起点。

* **核心功能**: 商品详情页
* **优先级**: P0 (核心必做)
* **业务原因阐述**: 消费者在下单前必须查看详尽的商品信息（价格、库存、SKU 规格）以做出购买决策。

### 2.3 交易履约
* **核心功能**: 购物车
* **优先级**: P0 (核心必做)
* **业务原因阐述**: 支持多件商品合并结算，是电商核心交易链路的重要组成部分。

* **核心功能**: 提交订单页
* **优先级**: P0 (核心必做)
* **业务原因阐述**: 结算流程的核心枢纽，涉及库存的锁定校验与最终支付金额的核算。

* **核心功能**: 支付模块
* **优先级**: P0 (核心必做)
* **业务原因阐述**: 商业变现的关键节点，无支付通道则交易无法最终成立。

* **核心功能**: 订单列表与详情
* **优先级**: P0 (核心必做)
* **业务原因阐述**: 用户查阅购买记录、追踪物流状态以及获取售后凭证的必经路径。

### 2.4 体验优化与逆向流程
* **核心功能**: 商品搜索 / 分类筛选
* **优先级**: P1 (重点提升)
* **业务原因阐述**: 提升用户在海量商品中寻找目标的效率。初期非阻断主流程需求。

* **核心功能**: 取消订单 / 退款申请
* **优先级**: P1 (重点提升)
* **业务原因阐述**: 属于逆向交易流程。MVP 初期可通过人工客服介入处理，系统化功能可延后。

## 3. 运营/管理端 (Admin Side) 功能矩阵

本模块旨在保障 B 端供给与履约交付。

### 3.1 商品供给与履约交付
* **核心功能**: 商品与库存管理
* **优先级**: P0 (核心必做)
* **业务原因阐述**: 后台必须有商品录入和可用库存控制机制，前端才能有货可卖。

* **核心功能**: 订单与发货管理
* **优先级**: P0 (核心必做)
* **业务原因阐述**: 商家必须能够查收已支付订单并录入物流进行发货操作，以完成交易闭环。

### 3.2 运营辅助与数据分析
* **核心功能**: 用户信息管理
* **优先级**: P1 (重点提升)
* **业务原因阐述**: 早期可查阅底层数据库，提供简易后台可提升客诉处理效率。

* **核心功能**: 核心数据看板
* **优先级**: P2 (后续迭代)
* **业务原因阐述**: 初期可通过手工导出数据统计分析，自动化数据看板留待后续迭代。
