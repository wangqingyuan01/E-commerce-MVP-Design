# 消费者下单与支付流程 (Order & Payment Flow)

## 1. 流程设计概述
本流程涵盖了电商系统最核心的资金流与数据流转，是前台交互与后台服务耦合度最深的模块。设计重点在于保障高并发场景下的数据一致性，防范超卖风险，并妥善处理第三方支付网关的异步状态同步。

---

## 2. 正常业务流 (Normal Flow)

### 业务逻辑可视化 (Visualized Flowchart)
![电商交易链路流程图](../assets/flowchart.png)

> **逻辑说明**：上图梳理了从用户发起结算到第三方支付回调的全链路逻辑，重点标注了库存预扣与实扣的触发时机。
此路径描述用户顺利完成交易的理想状态：

1. **发起结算**: 消费者在商品详情页点击“立即购买”或在购物车勾选商品后点击“结算”。
2. **前置强校验**: 系统服务端拦截请求，对商品当前上下架状态、可用 SKU 库存水位以及价格进行实时强校验。
3. **创单与预扣库存**: 校验通过后，系统在数据库中生成唯一订单号，并执行“预扣库存”（锁定库存）操作，防止被其他并发请求买空。
4. **唤起收银台**: 系统计算最终应付金额，唤起第三方支付网关（如微信/支付宝），消费者输入密码提交支付。
5. **异步回调确认**: 服务端接收到支付网关的异步成功回调通知，校验金额无误后，执行“实扣库存”，并向下游仓储物流系统分发履约指令。

## 3. 异常边界处理 (Exception Scenarios)

业务逻辑的严密性体现在对异常分支的阻断与引导：

### 3.1 购物车商品失效拦截 (Product Off-sale)
* **触发条件**: 用户在购物车停留时间过长，期间某商品被运营人员在后台强制下架。
* **处理机制**: 提交结算 API 返回失效状态，前端阻断支付跳转，并在页面顶部弹出警示 Toast，提示用户“部分商品已下架或失效，请修改购物车后重试”。

### 3.2 并发抢单与防超卖 (Insufficient Inventory)
* **触发条件**: 多个用户同时对仅剩 1 件的 SKU 发起结算。
* **处理机制**: 订单提交请求到达数据库层时，基于唯一约束或乐观锁机制，仅最先到达的请求能够成功锁定库存。后续落后的请求将被直接驳回，前端抛出“手慢了，该商品库存不足”的异常提示。

### 3.3 支付异常中断 (Payment Failure)
* **触发条件**: 因银行卡限额、网络抖动或用户主动关闭收银台导致支付未完成。
* **处理机制**: 订单保留在系统中，状态流转为待支付。在订单详情页提供“继续支付”的重试入口。

### 3.4 支付超时熔断 (Payment Timeout)
* **触发条件**: 订单创建后，用户在系统规定的生命周期内（例如 30 分钟）未完成资金结算。
* **处理机制**: 系统通过延时队列或定时任务主动扫描超时订单，将其强行关闭。**关键动作**：必须同步释放前期锁定的 SKU 库存，使数值回滚，供其他真实买家购买。

---

## 4. 核心状态流转节点 (Status Changes)

订单状态的变更必须遵循严格的单向流转逻辑，本流程涉及的核心节点跃迁如下：

* **初始创建**: `PENDING_PAYMENT` (待支付)
* **顺利成交**: `PENDING_PAYMENT` $\rightarrow$ `PAID` (已支付)
* **超时或主动放弃**: `PENDING_PAYMENT` $\rightarrow$ `CANCELLED` (已取消)

---
> **💡 产品架构与技术落地思考 (Tech-Driven PM Note)**
> 在规划下单流程时，我重点考虑了系统底层架构的数据安全性。在预扣库存与创单环节，建议研发团队利用 MySQL 等关系型数据库的事务控制 (Transaction) 确保操作的原子性。同时，在扣减库存的 SQL 语句中引入 `stock = stock - buyCount WHERE stock >= buyCount` 的乐观锁思想，可以最大程度地在数据库物理层兜底，彻底杜绝高频并发下的超卖事故。
